# ADR-001 — Tratar el código existente como spike de referencia y reconstruir test-primero

- **Estado:** Aceptada
- **Fecha:** 2026-06-08
- **Decisores:** Fabian (producto/arquitectura), Agente Analista
- **Fase del pipeline:** Transición Fase 0 → Fase 1

## Contexto

Una auditoría del repositorio (2026-06-08) encontró un prototipo funcional de
TranscriptorPy: la CLI transcribe archivos reales y exporta texto/JSON, existe
un esqueleto de FastAPI y un modelo de dominio embrionario (`TranscriptResult`,
`TranscriptSegment`). Sin embargo:

- 0 de 12 artefactos del pipeline obligatorio están presentes (sin `CLAUDE.md`,
  sin spec formal, sin features, sin ADRs, sin schemas).
- La cobertura de tests es <20%: solo `utils.py` está cubierto; el núcleo
  (`transcriber.py`, `exporters.py`, `api.py`, `cli.py`) no tiene tests.
- El código viola dos reglas innegociables del pipeline:
  1. Fue escrito sin test-primero.
  2. Eligió stack (faster-whisper, PyAV, FastAPI) ANTES de existir un solo
     escenario Gherkin.
- Además, diverge de las decisiones cerradas en Fase 0:
  - Renderiza marcas de tiempo; la salida acordada es texto corrido sin tiempos.
  - Exporta JSON; no está en la salida de v1 (.txt + documento).
  - No maneja URLs (YouTube), que sí están en el alcance de v1.

El código existe como herramienta funcional, pero no como proyecto gobernado.

## Decisión

1. **Congelar** el código actual como **spike de referencia de solo lectura.**
   Sirve como prueba de viabilidad y como insumo de conocimiento (qué se probó
   que funciona), no como base de la implementación.
2. **Reconstruir** TranscriptorPy desde Fase 1 siguiendo el pipeline completo:
   spec formal → Gherkin → TDD (Red-Green-Refactor) → arquitectura emergente →
   refactor continuo.
3. **No heredar el stack del spike.** Las tecnologías (motor de transcripción,
   extracción de audio, framework web, etc.) se decidirán DESPUÉS del Gherkin y
   se ratificarán en un ADR posterior. El spike puede informar esa decisión,
   pero no la predetermina.

## Consecuencias

### Positivas
- El pipeline queda íntegro: ninguna regla queda rota desde el inicio.
- El dominio y la arquitectura emergen de los tests, no de un prototipo apurado.
- Se evita arrastrar deuda técnica que ya no aplica (timestamps, salida JSON).
- La decisión de stack se toma con información, no por inercia del prototipo.

### Negativas / costos
- Se reescribe código que ya funciona: hay reescritura deliberada.
- Puede sentirse como "retroceso" frente a un PoC que ya transcribe.
- Hay que re-validar la viabilidad técnica bajo el enfoque test-primero.

### Mitigaciones
- El spike se conserva (no se borra) como referencia consultable.
- Lo que el spike probó (p. ej. que un motor de transcripción local produce
  texto utilizable) entra como insumo al definir el stack después del Gherkin.

## Alternativas consideradas

- **Caracterizar y refactorizar (descartada):** escribir tests de
  caracterización sobre el código existente y refactorizar desde ahí. Rechazada
  porque ratifica el stack pre-Gherkin y normaliza el haber escrito código sin
  test-primero.
- **Híbrido / rescate parcial (descartada):** rescatar piezas concretas bajo
  test y descartar el resto. Rechazada por simplicidad y para no arrastrar
  decisiones de diseño del prototipo al dominio nuevo.
